1. Product Overview

1.1 Vision

PoolDN is an in-house mobile-first web application for managing pool and billiards competitions, leagues, and community. It gives organizers the tools to run structured tournaments, teams the ability to apply and compete, and players a community hub to track their progress.

1.2 Problem Statement

Local billiards leagues are typically managed through spreadsheets, group chats, or manual tracking — leading to scheduling headaches, disputed results, and poor visibility into standings. PoolDN replaces this with a dedicated platform purpose-built for the sport.

1.4 Key Product Principles

  • Mobile-first — designed for phones first, responsive on desktop
  • Dark theme throughout for a modern, sports-app feel
  • Geolocation-aware — competitions, teams, and players are city-linked
  • Action-driven lifecycle — competitions progress through explicit organizer actions

3. MVP Scope Definition

The current v1.0 build is feature-rich but contains complexity beyond what is needed to validate the core product. Below is a recommended trimming of the scope to a true MVP — focused on the single most valuable loop: running a team-based round-robin pool league.

3.1 The Core Loop

  • Organizer creates and publishes a competition
  • Team captains apply and are accepted
  • Matches are auto-generated
  • Results are submitted after each match
  • Standings update and are visible to everyone

4. Core User Flows (MVP)

4.1 Competition Creation & Management Flow

Updated Mar 9, 2026 — Full 13-step flow defined; venue mode, game structure, Close & Schedule action, player rating formula documented

Primary flow for the Competition Organizer. Teams-only, Round Robin/League, MVP scope.

#StepDetail
1New CompetitionOrganizer clicks 'New Competition' button.
2Create formModal/form with fields: Competition Name (text), Game Type (8-ball/9-ball/10-ball — informational only, no logic impact), Format (Teams — MVP only), Tournament Type (Round Robin / League — MVP only), Start Date (informational), Prize Pool (sum of prize money — informational). Submitting creates a Draft competition.
3Draft createdDraft competition is saved but not publicly visible. Organizer can leave and return later to complete setup. Status: Draft.
4Tab: ParticipantsParticipation rule: Anyone can apply (MVP — invite-only is Phase 2). Team size: min and max number of players per roster. Teams must meet this requirement to apply.
5Tab: StructureOrganizer builds an ordered list of game slots for each matchday. Each slot is either a game (type: Singles or Doubles) or a break (with a duration). Example: Game 1: Singles, Break: 10 min, Game 2: Doubles, Game 3: Singles. This structure is the template used when captains assign players and submit results during active matches.
6Tab: ScheduleVenue mode: Central Venue (one fixed location, address entered here) or Team Home Venues (each team provides their home venue after being accepted). Games per Opponent: how many times teams face each other (1 = single round, 2 = home & away). Scheduling type: Weekly Rounds — organizer selects weekdays and a match time; matchdays are auto-generated based on team count and selected days. Note: if two teams share the same home venue, the system ensures one plays away on that matchday.
7Tab: Review & PublishFull preview of competition rules and terms. This exact content becomes the 'About' tab visible to all users after publishing. Organizer clicks 'Publish'.
8Published — UpcomingStatus changes to Upcoming. Competition is now publicly visible. Teams can browse and apply. Competition detail shows tabs: About, Teams, Players.
9Tab: AboutPublic-facing rules and terms from the Review step. Visible to everyone. Apply button shown to eligible teams.
10Tab: TeamsVisible to all but application management actions (accept/reject) shown to organizer and admins only. Pending applications listed here. The 'Close Applications & Schedule Matches' action block lives in this tab (organizer/admin only).
11Tab: PlayersIndividual player rating table across all accepted teams.

Rating = Points Earned ÷ Points Available × 100

Singles win = 3 pts, Doubles win = 2 pts, loss/appearance = 0 pts. Example: player plays 2 singles + 1 doubles, wins both singles — earns 6 pts out of 8 available = 75% rating. Formula fixed for MVP; configurable in Phase 2.
12Close Applications & Schedule MatchesSingle button action (organizer/admin only). Triggers: all pending/non-accepted team applications are automatically rejected with notification; Berger round-robin algorithm generates the full match schedule based on accepted teams, selected weekdays, match time, and venue mode (with home/away conflict resolution); competition status changes to Active.
13Active — OngoingMatches are scheduled and visible. Team captains or Roster Leaders assign players to game slots per the Structure template and submit results game by game. Standings and player ratings update after each match. No new applications accepted.

5. Technical Architecture

5.1 System Overview

PoolDN uses a three-tier architecture: a React single-page application on the client, a Hono web server running on Supabase Edge Functions (Deno runtime) as the backend, and a Supabase-hosted PostgreSQL database for persistence.

TierDetails
Frontend (Client)React SPA, TypeScript, Tailwind CSS v4. Communicates with backend via REST over HTTPS with Bearer token auth.
Server Layer (Edge)Hono framework on Deno runtime, deployed as a Supabase Edge Function. Handles all business logic and auth validation. Routes prefixed: /make-server-*/
DatabasePostgreSQL via Supabase. Currently implemented as a key-value store (single table: kv_store). All entities stored as JSONB blobs. Auth handled by Supabase Auth.

5.2 Key Data Entities

  • Player — user profile linked to Supabase Auth
  • Team — named group with a captain and member list
  • Competition — the main event entity, with type/format/status and draftSetup config
  • TeamParticipation — join table between Team and Competition, tracks application status and roster
  • TeamInvitation — captain-to-player invitation for team membership
  • Match — individual match record with scores, schedule, and player assignments
  • Notification — in-app notification per user with type, message, and action link
  • Venue — venue directory entry linked to competitions and teams

5.3 Authentication

  • Auth is handled by Supabase Auth (email + password)
  • Session token (JWT) stored in localStorage, sent as Bearer token on all API calls
  • Server validates token with Supabase on every protected request
  • Player profile created automatically on signup

5.4 Competition Lifecycle — Technical

The competition lifecycle is action-driven: organizer clicks trigger state transitions via POST endpoints.

  • POST /competitions/:id/publish — Draft → Upcoming
  • POST /competitions/:id/close-applications — locks applications
  • POST /competitions/:id/configure-schedule — stores weekdays + match time
  • POST /competitions/:id/generate-matches — runs Berger algorithm, creates Match records, cleans up pending data, sets status → Active

6. Known Limitations & Technical Debt

6.1 Critical: Key-Value Store Architecture

The current database layer stores all data as JSON arrays in a single PostgreSQL table (kv_store). While this accelerated initial development, it introduces significant limitations for production use:

  • No transactions — concurrent writes can corrupt data
  • Full array read/write on every operation — poor performance at scale
  • No native querying, filtering, or indexing
  • No referential integrity — cascading deletes must be managed manually in application code
  • No database migrations path — schema changes require manual data transformation
⚠️ Recommendation: Migrate to a proper relational schema (separate tables per entity with foreign keys, indexes, and Supabase Row Level Security) before scaling beyond a pilot. This is the highest-priority technical debt item.

7. Roadmap

Phase 1 — True MVP

Updated Mar 9, 2026 — Scope updated to reflect confirmed MVP decisions

Ship a focused, stable version covering the core league loop. Cut complexity, fix critical infrastructure gaps, validate with real users.

  • Scope: Auth, Round Robin / Teams / 8-ball competitions with full 4-tab draft setup
  • Prize pool field on competition creation
  • Customizable ordered game structure (game slots + break times) per competition
  • Venue mode: central venue or team home venues, with home/away conflict resolution in scheduling
  • Weekly rounds scheduling with Berger algorithm match generation
  • Roster Leader role: captain appoints a stand-in when not joining the competition roster
  • Competition player ratings: Rating = Points Earned ÷ Points Available × 100 (Singles win = 3 pts, Doubles win = 2 pts, loss/appearance = 0 pts)
  • Competition detail tabs: About, Teams (with organizer tools), Players (rating table)
  • Basic in-app notifications: team invites, application status, match results
  • Tech: Migrate database to proper relational schema (critical pre-launch dependency)
  • Ops: Configure SMTP, tighten CORS, add basic error monitoring
  • Goal: Run one real pilot league end-to-end

Phase 2 — Feature Expansion

Once MVP is validated, expand based on user feedback:

  • Single Elimination and Double Elimination tournament types (with bracket visualization)
  • Doubles and Mixed match formats within team competitions
  • 9-ball and 10-ball game type support
  • Fixed Matchdays scheduling (central venue, multiple games per day — suited to elimination events)
  • Invite-only competition mode
  • Venue management as a standalone module (venue profiles, directory)
  • Community / player directory
  • City / geolocation-based filtering
  • Proof image upload for match result verification
  • Customizable player rating formula (configurable in competition settings)
  • Email notifications

Phase 3 — Growth & Platform

Platform features to drive retention and scale:

  • Real-time match updates (Supabase Realtime)
  • Push notifications (mobile)
  • Player ranking system
  • Advanced statistics and analytics
  • Tournament bracket visualization
  • In-app chat / messaging
  • Calendar integration
  • Sponsorship and prize pool management
  • Multi-language support
  • Native mobile app (React Native)
  • Admin dashboard and operational tools
